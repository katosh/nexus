# External dependency surface

> **Purpose.** A complete, grep-able inventory of every external interface nexus-code depends on — CLI flags, env vars, parsed TUI strings, API endpoints and fields, subcommands, settings keys — with a *breakage signal* for each. It exists so that when an upstream tool (above all **Claude Code**) ships a changelog or news item, an automated process can map that change to the exact nexus-code `path:line` it could break.

This document catalogs **385 interfaces** across **11 dependencies**. Completeness is the design goal: a missed interface is a blind spot the automation can never flag, so the inventory errs toward over-inclusion. Entries were gathered by reading every shell script, settings file, config, hook, and CI workflow in the repo and spot-verified against the source.

## How to use this doc (update-impact workflow)

Given an upstream changelog / news / release-note item:

1. **Identify the tool.** Which dependency does the item concern? Jump to its section via the [index](#dependency-index) below. Claude Code is split into four subsections (CLI, env/settings, **TUI strings**, session model) because it is the highest-churn dependency.
2. **Scan the *Breakage signal* column.** Each row states what *kind* of upstream change would break that interface ("spinner glyph changed", "`--json` field renamed", "hook event renamed", "status-bar layout reflowed"). This is the column the changelog-matcher keys on — match the news item against these phrases.
3. **Open the *Where* `path:line`(s)** for any matched row and assess the actual code. The locations are exact and clickable.
4. **Check test coverage.** For Claude Code interfaces, the [test-coverage table](#claude-code-test-coverage) marks each as COVERED (with the test/fixture that pins it) or **UNTESTED**. Untested fragile interfaces are where a silent break is most likely — prioritize those.

Every section uses the **same four-column schema** so the doc is uniformly machine-parseable:

| Column | Meaning |
|---|---|
| **Interface** | The specific flag / env var / parsed string-or-glyph / endpoint / field / subcommand. |
| **Where** | One or more `path:line` locations (repo-root-relative). |
| **Purpose** | What nexus-code uses it for. Entries tagged `(test)` are test-only usages. |
| **Breakage signal** | The class of upstream change that would break it — the matcher key. |

!!! warning "Fragility ranking"
    **Claude Code** is the most fragile dependency: it parses Claude Code's *rendered terminal output* (spinner glyphs, ANSI colour codes, status-bar wording, overlay text) to classify worker state. A Claude Code release that re-themes the TUI, renames a hook event, changes a CLI flag, or relocates the session log can silently break state detection with no error. The [TUI / pane-rendering section](#3-claude-code-tui-pane-rendering-string-parsing) is the single most important part of this doc to keep current.

!!! note "Notes"
    VERSION PINNING: package.json declares @anthropic-ai/claude-code 2.1.173 as a MAINTAINER-MANAGED VETTED FLOOR for fresh installs (not a per-release pin); package-lock.json locks node_modules to 2.1.161 (the resolved version for reproducible npm ci). The two intentionally differ: the floor advances only when a maintainer vets a release, while a successful gated cc-update advances the operator-LOCAL pin (monitor/.state/cc-version-local, gitignored) without editing either tracked file. install-claude-local.sh implements an idempotent fast-path: if node_modules/.bin/claude exists and reports version == pinned_version, skip reinstall; else npm ci (reproducible with lock) or npm install (first-time setup). Requires Node.js >= 18.0.0. See monitor/_cc-version.sh and skills/nexus.cc-update/GUIDE.md.
    
    ENVIRONMENT SUMMARY: Three main env vars exported into Claude processes: NEXUS_ROOT (repo path), NEXUS_IS_ORCHESTRATOR (0 or 1, orchestrator only), NEXUS_WORKER_WINDOW (tmux window name, worker only). Plus CLAUDE_CODE_RESUME_THRESHOLD_MINUTES and CLAUDE_CODE_RESUME_TOKEN_THRESHOLD (respawn paths only, to suppress resume dialog).
    
    SETTINGS FILES: Two JSON files passed via --settings: orchestrator-settings.json (orchestrator only, includes session-pin hook) and worker-settings.json (worker only, includes heartbeat/notification/decision-emit hooks). Both set skipDangerousModePermissionPrompt: true and an env block with DISABLE_AUTOUPDATER=1 (the built-in npm auto-updater never applies to the project-local pin; disabling it removes the spurious "Auto-update failed" banner).
    
    HOOK SCHEMA: Hooks are keyed by event (PreToolUse, PostToolUse, Notification, UserPromptSubmit, PermissionRequest, Stop, StopFailure) and optional matcher (AskUserQuestion, Bash, Write, Edit, NotebookEdit). Each hook runs a shell command that typically redirects jq output into state files under $NEXUS_ROOT/monitor/.state/.
    
    SESSION RESUMPTION: The watcher owns it via the orchestrator session-id pin: _respawn_choose_resume_mode resolves a valid pin to --resume <uuid> and degrades to a deterministic cold fresh spawn otherwise (never bare --continue — issue 200). entry.sh only reconciles operator intent onto the pin (default archives it; --continue keeps it). claude-loop.sh's --continue path sets CLAUDE_CODE_RESUME_THRESHOLD_MINUTES=999999999 to suppress the resume dialog. The pin file is written at spawn and re-affirmed on every UserPromptSubmit via orchestrator-session-pin.sh hook.
    
    BINARY RESOLUTION: _claude-bin.sh is sourced by all spawn paths. Resolves CLAUDE_BIN in order: env override, project-local $NEXUS_ROOT/node_modules/.bin/claude, system `claude` on PATH. Path is baked into launcher heredocs at write time so spawned windows use absolute path, not PATH lookup.
    
    TEST FILES: test-*.sh files in monitor/watcher/ test spawn-worker.sh, spawn-fresh-orchestrator.sh, and respawn logic. Tests capture launcher body and assert on presence of --dangerously-skip-permissions, --continue (default), --resume (when pin valid), --settings (when file exists). Not included in the entries above but should be scanned for breaking regressions if Claude Code CLI changes.
    
    GAPS / ASSUMPTIONS: (1) Hook payload schema is inferred from jq filters in orchestrator-session-pin.sh and worker hooks; no formal schema doc found. (2) session_id field name assumed from orchestrator-session-pin.sh jq '.session_id'; if Claude Code renames it, the hook silently fails. (3) Project slug encoding (char → dash) is documented in _respawn.sh but not verified against Claude Code's actual implementation. (4) Resume dialog suppression assumes CLAUDE_CODE_RESUME_* env vars still exist in Claude Code; if removed or renamed, the respawn paths will fail silently and the picker will block.

!!! note "Notes"
    **Version Assumptions and Compatibility Notes:**
    
    1. **GitHub REST API Version (2022-11-28)**: Hardcoded in curl calls and gh CLI invocations. If GitHub retires this version, header X-GitHub-Api-Version must be updated across monitor/mint-token.sh, monitor/watcher/_deliveries.sh, and monitor/user-pat.sh. The gh CLI may auto-negotiate a newer version; curl calls will fail outright if the header is no longer accepted.
    
    2. **GraphQL Schema Stability**: GraphQL queries in _github.sh and _mentions.sh rely on exact field names and structure. Breaking changes include: (a) removing `databaseId` (deprecated in favor of `id` on many objects), (b) changing `comments(last: N)` pagination pagination, (c) renaming `reactions.nodes[].content` enum values (EYES/ROCKET), (d) removing `.repository.nameWithOwner` in favor of `.repo` or similar, (e) removing `reviewThreads` collection on PR objects. The per-surface backoff and gate mechanisms (_graphql_backoff_active, _graphql_polling_gate, _watcher_handle_graphql_failure) are resilient to some transient failures but will fail outright on schema changes.
    
    3. **Installation Token Lifecycle**: POST /app/installations/<id>/access_tokens is the JWT→token exchange. Breakage scenarios: (a) endpoint path changes, (b) JWT format rejected (currently RS256 with iss/iat/exp), (c) response .token/.expires_at fields missing, (d) expiry format incompatible with date -d (currently ISO 8601). The 55-minute cache window (300 s safety margin) assumes expires_at is reliably 1 hour in the future; if GitHub changes the TTL, cache hits/misses will become stale.
    
    4. **Webhook Delivery Payload Structure**: _deliveries.sh parses .event and .action enums (e.g., issue_comment, pull_request_review_comment, created, submitted, opened) to gate emission. If GitHub changes these enum values, the conditional logic on line 313 onward (case "$event" ... elif ...) will silently drop events and surface nothing, even though the API call succeeds. Similarly, field paths like `.request.payload.issue.number` and `.comment.user.login` are hardcoded; reorganization (e.g., `.issue` → `.issue_data`) breaks silently.
    
    5. **Reaction Content Enum (EYES, ROCKET)**: Hardcoded in ng verb bodies and GraphQL jq filters. If GitHub renames these enums (e.g., "EYES" → "EYE", "ROCKET" → "ROCKET_EMOJI") or removes them, reactions will silently POST invalid content (4xx error) or fail to filter incoming reactions (no matching .content value). The _functional_check helpers that test reactions assume the enum values are stable.
    
    6. **Rate Limit Bucket Structure**: _graphql_polling_gate expects `.resources.graphql.remaining` from /rate_limit. If GitHub flattens the structure (e.g., `.graphql.remaining` instead of `.resources.graphql.remaining`), the jq extraction fails silently and the gate assumes bucket is healthy, potentially drowning in rate-limit errors. The probe also assumes the free REST core bucket (used for /rate_limit itself) is separate from the GraphQL bucket; convergence or deprecation of one will degrade the gate's utility.
    
    7. **Cross-Repo Context**: snapshot_mentions and deliveries (mention= shape) rely on `.repository.nameWithOwner` to disambiguate cross-repo events. _filter_cross_repo_surface makes routing decisions based on whether the emit header contains `mention=` or `cross_repo=` to distinguish App-installed repos (mention, can comment) from search-only hits (cross_repo, cannot comment). If GitHub changes how these contexts are signaled in webhooks or search, the downstream routing and capability model breaks.
    
    8. **No GraphQL Mutations**: The watcher is read-only on the GraphQL side. All writes (reactions, comments, issue/PR updates) go through REST. If GitHub moves reactions to GraphQL-only or deprecates the REST endpoints, substantial refactoring is needed.
    
    9. **Test Infrastructure**: Many test files (test-emit-gate.sh, test-graphql-gate.sh, etc.) mock `gh` by shadowing it on PATH with a shell stub. These mocks hardcode response shapes (e.g., `{"graphql":{"remaining":5000,...}}`). Tests will fail immediately if response shapes change, but production code may not fail until that code path is actually executed (e.g., a new edge case in the snapshot logic).
    
    10. **Pagination Cursor Stability**: monitor/.state files (`last-mention-cursor.txt`, `last-delivery-cursor.txt`, `processed-comments.txt`) store comment databaseIds, delivery GUIDs, and issue numbers. If GitHub changes how these identifiers are minted, assigned, or persisted (e.g., databaseId format changes from 32-bit to 64-bit int), the cursor-based dedup logic may skip comments or re-emit duplicates.
    
    11. **HTTP Status Codes**: Code assumes 200 = success, 404 = not found (webhook not configured), 401 = auth failure. If GitHub changes status semantics (e.g., 404 → 410 Gone for /app/hook/deliveries when webhook is unconfigured), the handling may silently degrade or error differently.
    
    12. **Completeness**: This inventory covers every `gh api`, `gh auth`, `curl -H Authorization: Bearer`, GraphQL query field, and REST endpoint used in production watcher and ng code. Test-only usages are tagged `(test)`. The inventory is per-file-location; if code is refactored across files or files are deleted, the paths will need updating but the interface dependencies remain the same.

!!! note "Notes"
    ## Version Assumptions
    
    - **jq**: Tested against jq 1.5+ (as noted in comments about IEEE 754 number parsing). No version check enforced; degradation is silent with grep fallback.
    - **python3**: >=3.6 (implicit from f-string and type hint usage in embedded scripts). Requires pyyaml for YAML parsing.
    - **Node.js**: >=18.0.0 (enforced in install-claude-local.sh line 52-54). npm required for package-lock.json handling.
    - **bash**: 4.0+ minimum for associative arrays and [[ ]]; 4.3+ for nameref (local -n).
    - **GNU tools**: sed -E, grep -P, stat -c, date -d, readlink -f are GNU-specific (no BSD fallback). System must be Linux or provide GNU coreutils.
    - **SLURM**: Assumes sbatch and srun output format is stable (Submitted batch job N, Submitted job N). Patterns in async-launch-patterns.conf are data-driven and can be updated without code changes.
    - **agent-sandbox**: Required for safe isolation. entry.sh and bootstrap-install.sh fail hard if SANDBOX_ACTIVE != 1.
    
    ## Gaps and Ambiguities
    
    - **coreutils version pinning**: No explicit minimum version for individual tools. Assumption is Linux with GNU coreutils from the last 5-10 years.
    - **tmux**: No version check; assumes tmux 2.0+ based on option syntax. Window name restrictions and ANSI handling may vary by version.
    - **jq line-by-line reads**: Some filters are applied to single-line JSON (heartbeat files); behavior depends on jq's handling of compact JSON. Multiline JSON in stdout/stdin is capped at 64KiB by design.
    - **Python imports**: Only yaml is imported in load.sh; json/sys imports are present in test files but implicit.
    - **Regex flavor**: bash [[ ]] uses POSIX ERE; grep -P uses PCRE (gnu-only). No attempt to support other flavors.
    - **Numeric precision**: jq comment at pane-state.sh line ~1025 notes IEEE 754 parsing; large epoch times may lose precision on some systems.
    - **Date handling**: Assumes TZ environment variable is respected; some date -d formats may not work across all locales.
    
    ## Files Skipped
    
    - test fixture JSON files (monitor/watcher/fixtures/*.json) — contain test data, not functional dependencies.
    - .github/workflows/*.yml — GitHub Actions CI definitions; do not affect runtime.
