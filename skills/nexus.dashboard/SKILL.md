# nexus.dashboard ‚Äî overview issue identity block + dashboard schema

TRIGGER when: setting up a new operator nexus and seeding the overview
issue `#1`; the working directory / host / repo provenance of a nexus
needs recording somewhere notable; the dashboard's content has drifted
between runs and needs a stable skeleton; reviewing or editing what
`ng dashboard put` pushes; an operator (or fresh orchestrator) asks
"where does this nexus live / how do I orient."

## The two rules

1. **Identity lives in `#1`, auto-generated.** The working directory
   and machine identity of a nexus belong in the overview issue body
   (`nexus:overview`, conventionally `#1`) ‚Äî the one notable surface
   every operator sees ‚Äî as an **auto-generated, delimited block**, not
   hand-typed prose. Generate it with `ng nexus-identity`.

2. **The dashboard has a fixed schema.** Every operator's dashboard
   uses the **same six sections**, scaffolded and validated by
   `ng dashboard scaffold` / `ng dashboard validate`. The section set is
   the single source of truth (`DASH_REQUIRED_SECTIONS` in `monitor/ng`)
   ‚Äî change it there and it propagates to every nexus on the next
   `nexus-code` pull.

Both are **standard across all operator nexuses.** They are derived
entirely from each nexus's own environment/config (`$NEXUS_ROOT`,
`github.repo`, hostname, the watcher pidfile/log paths) ‚Äî nothing is
hardcoded to one operator ‚Äî so a second operator running the same code
gets their own correct identity block and conforming dashboard with
zero edits.

## Identity block ‚Äî `ng nexus-identity`

```
ng nexus-identity --dry-run          # render to stdout, no GitHub write
ng nexus-identity --upsert-overview  # idempotently PATCH the block into #1
```

The block is bounded by `<!-- nexus-identity:start -->` /
`<!-- nexus-identity:end -->` and is upserted in place (replace, never
duplicate) ‚Äî exactly the precedent set by `ng interactive-sessions`.
Repeated runs converge; it can't drift. The bot identity (installation
token, REST PATCH) is used for the write.

Fields, all **derived** (nothing hardcodes `<operator>`):

| Field | Source |
|-------|--------|
| **Working directory** (headline) | `$NEXUS_ROOT`, else script-relative root |
| Host | `hostname` |
| Asset + issue repo | `github.repo` + overview issue number |
| Implementation clone | the nexus root's git `origin` remote + current branch |
| Watcher pidfile / log | `<root>/monitor/.state/watcher.{pid,log}` |

It is **identity, not status** ‚Äî a stable "where things live" record.
Live status (what's running, what's blocked) is the dashboard's job,
below it in the same issue body. The block ends with a one-line
orient/resume pointer (cockpit `monitor/svc.sh`, dashboard location,
reports dir) and a "do not edit by hand ‚Äî auto-generated" note.

Who runs it: the **orchestrator**, from the primary clone, after a
fresh setup or whenever the root/host/repo provenance changes (rare).
It's cheap and idempotent, so re-running it to repair drift is safe.

## Dashboard schema ‚Äî `ng dashboard scaffold | validate | put`

The dashboard is the content **between** the
`<!-- NEXUS_DASHBOARD_START/END -->` markers in `#1`. It must carry
these six H2 sections, in this order:

| Section | Populated by |
|---------|--------------|
| `## Identity` | **Pointer** to the `ng nexus-identity` block above ‚Äî never a copy (single source of truth) |
| `## Infra` | Operator-narrated: watcher / orchestrator / CC-pin health |
| `## Services` | Auto-populatable from `monitor/services.registry` (name + healthcheck); narrate exceptions |
| `## In-flight` | Active worker windows (mirrors `tmux list-windows` + heartbeats) |
| `## Awaiting operator` | Threads/decisions blocked on the operator |
| `## Recent landings` | Last few merged PRs / completed tasks |

Verbs:

```
ng dashboard scaffold                 # print the canonical skeleton (hints inline)
ng dashboard validate [--body-file P] # strict: exit 1 if any required section missing
ng dashboard put --body-file P        # splice + PATCH; WARNS on missing sections, never blocks
ng dashboard get                      # fetch the current dashboard middle
```

- **`scaffold`** emits the skeleton with a one-line hint per section.
  Seed a new dashboard by piping it into `put`, or use it to remember
  the section set.
- **`validate`** is the **strict gate** (exit 1 on drift) ‚Äî for CI /
  pre-commit / a deliberate conformance check.
- **`put`** runs the same check in **warn-only** mode: it prints which
  sections are missing to stderr but still pushes. Rationale: the
  operator must be able to update the dashboard in a hurry; a hard
  block there would be a footgun. Use `validate` when you want the
  hard failure.

## Consistency with the rest of the nexus

- The identity block is **metadata, not content** ‚Äî it does not
  violate the overview "routing-only" rule (`CLAUDE.md`). Per-task
  back-and-forth still belongs in dedicated issues; the identity block
  and dashboard are the standing skeleton of `#1`, not discussion.
- The schema-enforcement pattern mirrors `nexus.report`'s
  `ng report-check`: a canonical section list, a checker, a scaffold.
- Spawning, wrap-ups, and routing are unchanged ‚Äî see
  `nexus.tmux-spawn`, `nexus.report`, `nexus.bot`.

## Where the schema lives (for maintainers)

`monitor/ng`: `DASH_REQUIRED_SECTIONS` (the section array),
`_dashboard_template` (scaffold), `_dashboard_missing_sections`
(checker shared by `validate` and `put`-warn), and `cmd_nexus_identity`
(the identity generator + idempotent upsert). Tests:
`monitor/watcher/test-ng-dashboard.sh` (scaffold + validate + put-warn)
and `monitor/watcher/test-ng-nexus-identity.sh` (identity render +
idempotent upsert).
